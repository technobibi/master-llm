# 改善リスト（長期運用・OSS・ML基盤としての設計改善）

2026-07-13 の設計レビュー（外部レビュー＋実データ検証）で合意した改善方針。
**実装はまだ**。中核（router/スキーマ）は所有者の領域なので、ここは「決定済みの方針」の記録。
現行の計測基盤の正は [DESIGN-telemetry.md](DESIGN-telemetry.md)、ルーターは [DESIGN-router.md](DESIGN-router.md)。

## 背景（なぜ直すか）

- 実データで判明: **埋め込み単独では成否を分離できない**（NN一致率50%）。ただし**領域平均**なら
  成功率に57〜90%の差が出る兆候あり（[RESULTS.md](RESULTS.md) 2026-07-13）。
  → ベクトルは「規模＋仕様品質」と並ぶ1特徴であって主役ではない。
- 所有者の懸念（正しい）: Run は model×quant×hardware×runtime に依存するため、新モデルが
  出る度に再評価が要る。速度はハード依存で世界共有の価値が薄い。
  → **成果（ハード非依存・共有可）と性能（ハード依存・ローカル較正）の分離**が答え。

## 必ず直す（データモデルの根幹・安価で高効果）

1. **成果と性能を分離する**（最重要）
   - 成果（success / tokens）= (task × model × quant × harness)。ハード非依存 → 世界共有できる資産。
   - 性能（wall_s / tok_per_s）= (× runtime × hardware)。同一 hardware_id 内でのみ比較可 → ローカル較正。
   - ルーティング判定 = **P(成功|特徴,モデル)【全世界で学習・共有】 × コスト/速度【ローカル較正】**。

2. **`oracle_arm` を一次データとして保存しない**
   - 「成功>コスト>枠」という**政策依存の派生値**。政策が変わると正解ラベルが変わる。
   - 保存するのは per-arm の生スコア。判定は読み取り時に政策（版付き）を当てて導出する。

3. **success を二値でなく反復回数（n中m回）で持つ**
   - 温度>0 で成否は確率的。所有者の「過去n回中n回」構想と一致。確率として蓄積し、
     領域コンフィデンスの土台にする。

4. **ルーター特徴に規模＋仕様品質を足す**（埋め込み単独は弱いと実証済み）
   - 規模系: language / framework / repo_size / file_count / estimated_tokens。
   - **仕様品質系**（レビューが漏らした最重要）: 例(assert)の有無 / 型注釈の有無 / エッジケース記述。
     失敗の多くが能力でなく仕様欠陥だったため（mbpp_411/454/626）。

## OSS 化で必要になる

5. **Run の検証設計**（コミュニティ提出は偽造・再現不能）
   - Run の success は (model, quant, harness, seed) に依存し、再実行しないと信用できない。
   - 解: 再現可能ハーネス＋サンドボックス再実行、または「提出Runは参考値、権威は再実行のみ」と割り切る。

6. **タスクの賞味期限管理**（汚染対策）
   - 公開タスクは時間とともに学習データに混入する。task に作成日を持たせ、モデルの cutoff と
     突き合わせて汚染を判定。task の版管理と廃止フローを用意する。
   - 関連: content-addressed task_id（prompt+tests のハッシュ）で OSS の重複・ドリフトを検出。

## 設計判断として頭に置く（今すぐの実装ではない）

- **新モデルは全再走不要**: 多様性サブセット（farthest-point）で安く再特性化する。
  ← 埋め込みが真に役立つ場所（新モデルの安い特性化）。
- **採点の定義**が、コーディング以外へ広げる時の真のボトルネック。pytest の客観性は例外的に恵まれている。
  執行検証型（TOOLATHLON / Workspace-Bench）は今の設計にそのまま乗るが、翻訳・メールは連続値/審査型。

## 触らなくてよい（既に妥当）

- `run_id`（全Runにある）・`embedding_version`（task_vectors.jsonl の `model`）→ 既に実装済み。
- Embedding と Run の分離・JSONL+NumPy・ベクトルDB不要 → 現状のままで正しい（100万件規模で FAISS 等を検討）。
- append-only・1ファイル=1イベント種・生データ優先・schema版番号 → 既に DESIGN-telemetry の原則どおり。
