# 設計書: テストスイート（固定ベンチマーク）

**目的**: 同一のテストを繰り返し実行し、本システム（ルーター・ローカルモデル・構造化機構）の
**改善を時系列で計測できる**固定ベンチマークを定義する。
**状態**: 設計のみ（タスクの実装は別途）。

## 0. 大原則

1. **採点は静的・決定的**。AI採点（LLM-as-judge）は使わない。
   pytest・文字列/正規表現照合・DOM検査・色距離の4手段のみ。同じ解答は必ず同じ点になる。
2. **スイートは凍結してバージョン管理**。タスクの追加・変更をしたら `suite v2` として別名にする。
   バージョンをまたぐスコア比較はしない（ものさし自体が変わるため）。
3. **dev / holdout の分割**。ルーター等の学習に使ってよいのは dev タスクのみ。
   holdout はシステム改善の評価専用で、**学習データに決して入れない**
   （同じテストを繰り返して改善を測る以上、ベンチマークへの過適合=グッドハートの法則対策が必須）。
4. **課金モードはテスト設計と無関係**。サブスク/API の別は `CLOUD_BILLING` が
   ログの `cost_usd` / `api_equiv_usd` に記録するだけの計測ラベルであり、
   タスク・採点・実施方法には一切影響しない。
5. 既存の不変ルール（同一指示・隠しテスト・予算キャップ・反復3回・中央値）を継承する。

## 1. 採点方式（4種類・すべて静的）

| 方式 | 実体 | 使うカテゴリ |
|---|---|---|
| `pytest` | 隠し pytest の通過数 / 総数 | 簡単編集・機能追加・曖昧ペア |
| `report-match` | 解答ファイル（ANSWER.md 等）を正規表現・キーワード・パス照合 | 調査・バグ調査・脆弱性調査 |
| `ui-static` | Playwright での動作 assert + DOM 構造検査 + 色距離 | Web画面 |
| `manifest-recall` | 仕込みリスト（manifest）との照合で 発見数/仕込み数 | バグ・脆弱性（report-match の特殊形） |

### manifest-recall の判定規則（バグ・脆弱性共通）

- 仕込み1件 = manifest 1行: `{id, file, symbol（関数名）, line_range, type_keywords[]}`
- 報告書（BUGS.md / VULNS.md）中で「**file に言及 かつ（symbol または line_range 内の行番号に言及）**」
  で発見1件と数える。type_keywords（例: "SQL", "traversal|トラバーサル"）の一致で type 正解を判定
- 主指標 = **recall（発見数/仕込み数）**。manifest にない指摘は false_positive として別記録
  （成功判定には使わないが precision の推移を見る）
- manifest は `tests/` 内に置く（エージェントには渡らない）

### ui-static の3層採点（重みは 機能60 / 構造25 / 色15）

- **機能60%**: Playwright の動作 assert（クリック→状態変化 等）。1 assert = 1テスト
- **構造25%**: DOM 検査 — 要素の種類と数 / 出現順序 / 相対配置（要素の重心座標の上下左右関係）/
  サイズ比率（画面幅比、許容誤差 ±15%）
- **色15%**: 指定要素の computed style を取得し、正解サンプルとの **CIEDE2000 色差 < 10** で合格
- **ピクセル diff は主指標にしない**（フォント差・1pxズレで嘘の失点が出るため）。参考値として保存のみ
- 3層とも「N個の静的チェックの通過数」に落とすので、全体は tests_passed/tests_total に一本化できる

## 2. カテゴリと難易度マトリクス（suite v1 = 22タスク + UI変種）

| カテゴリ | 難易度 | 採点 | タスク数 | holdout |
|---|---|---|---|---|
| A. 簡単編集 | 低 | pytest | 3 | A3 |
| B. 調査（どこ?何?） | 低〜中 | report-match | 3 | B3 |
| C. バグ調査 | 中〜高 | manifest-recall | 3 | C3 |
| D. 脆弱性調査 | 中〜高 | manifest-recall | 3 | D3 |
| E. 機能追加・自立実行 | 中〜高 | pytest | 3 | E3 |
| F. 曖昧 vs 構造化ペア | 中 | pytest | 3ペア=6 | F3ペア |
| G. Web画面 | 中〜高 | ui-static | 3（+画像変種1） | G3 |

各カテゴリの3番目（*3）を holdout に固定。dev:holdout ≈ 2:1。

## 3. テストケース定義

### A. 簡単編集（ルーターが「ローカル行き」と判定すべき層）

- **A1 typo修正**（低）: seed の挨拶関数に "Helo, Wrold!" という文字列バグ。
  指示「タイポを直して」。pytest: 正しい文字列を assert（3テスト: 出力一致・関数名不変・他行不変）
- **A2 rename**（低）: `calc()` を `calculate_total()` に全参照込みで改名（呼び出し元2ファイル）。
  pytest: 新名で import 可・旧名が存在しない・挙動不変
- **A3 メッセージ日本語化**（低・holdout）: エラーメッセージ3種を指定の日本語に置換。
  pytest: 3メッセージの完全一致

### B. 調査（コードを編集せず ANSWER.md に回答を書かせる）

- **B1 関数の所在**（低）: 5ファイルの小さなアプリを渡し
  「入力バリデーションはどのファイルのどの関数にある?」。
  採点: ANSWER.md に `validators.py` と `validate_email` の両方が含まれる（2チェック）
- **B2 これは何をする関数?**（中）: 変数名が無意味な難読関数（実体はエラトステネスの篩）。
  「この関数は何をしているか説明せよ」。採点: キーワード照合（"素数|prime" 必須、
  "篩|エラトステネス|sieve" で加点、計3チェック）
- **B3 定義と使用箇所**（中・holdout）: 設定値 `MAX_RETRIES` の定義元と使用箇所2つを列挙させる。
  採点: 3つのファイルパスすべてに言及（3チェック）

### C. バグ調査（仕込み → BUGS.md に報告 → recall 採点）

- **C1 電卓カート**（中・仕込み3件）: ①合計計算のオフバイワン（`range(len(items)-1)`）
  ②割引の演算子逆（`+` が `-`）③可変デフォルト引数（`items=[]`）
- **C2 日付・文字列ユーティリティ**（中・仕込み5件）: ①タイムゾーン無視の比較
  ②`==` でなく `is` による文字列比較 ③貪欲すぎる正規表現 ④encoding 未指定の open
  ⑤ゼロ除算ガード漏れ
- **C3 キャッシュ付きAPIクライアント**（高・仕込み2件・holdout）:
  ①ミュータブルな共有 dict による状態汚染 ②TTL 判定の不等号逆転（期限切れを返し続ける）
- 指示は全ケース共通文面: 「このコードにはバグが N 件ある。BUGS.md に
  `ファイル:行` と説明を1件ずつ書け」（**件数 N は明かす**。探索の打ち切り条件を揃えるため）

### D. 脆弱性調査（防御目的の教材。実在サービスのコードは使わない）

- **D1 ログインAPI**（中・仕込み2件）: ①f-string による SQL 組み立て（SQLi）
  ②ソースにハードコードされた API キー
- **D2 ファイル配信サーバ**（中・仕込み3件）: ①パストラバーサル（`../` 未検査）
  ②管理ルートの認可チェック漏れ ③MD5 によるパスワードハッシュ
- **D3 URLプレビュー機能**（高・仕込み3件・holdout）: ①SSRF（内部IP未ブロック）
  ②pickle による外部データのデシリアライズ ③`subprocess(shell=True)` へのユーザー入力連結
- 指示・採点は C と同形式（VULNS.md、type_keywords 一致で種類判定まで見る）

### E. 機能追加・自立実行（隠し pytest。E2→E3 で複数ファイル・複数手順に）

- **E1 CSV集計**（中）: 既存CSVパーサーに「列を指定して groupby 合計」機能を追加。
  pytest 5本（正常系2・空入力・型混在・存在しない列のエラー）
- **E2 TODOアプリの永続化**（中高）: メモリ上の TODO CLI に JSON 保存/読込を追加。
  pytest 5本（保存→再起動→復元、空ファイル、壊れたJSONの扱い 等）
- **E3 ミニHTTPサーバ拡張**（高・holdout）: 標準ライブラリ製サーバに
  `/items` の GET（クエリでフィルタ）と POST（追加・バリデーション）を実装。
  pytest 6本（テストクライアントで正常系3・異常系3）。複数ファイル編集が必要な構成にする

### F. 曖昧 vs 構造化ペア（同一ゴール・同一隠しテスト・指示だけ2種）

> 将来の「ローカルLLMが曖昧指示を構造化してからクラウドへ渡す」機構（構造化arm）の
> 効果測定の土台。ペアの差分（Δ成功率・Δターン数・Δ出力トークン）が曖昧さのコスト。

- **F1 期限切れ表示**（E2 の TODO アプリに追加）:
  曖昧版「期限すぎたやつわかるようにして」/
  構造化版「`list_tasks()` の出力で、期限 < 今日 の行の先頭に `[期限切れ]` を付ける。
  期限なしタスクは対象外」。pytest は**中核解釈のみ**を検査（4本）
- **F2 ログ整形**: 曖昧版「ログ見やすくして」/
  構造化版「`format_log(record)` を `[LEVEL] 2026-01-01T00:00:00 message` 形式で返すよう修正。
  LEVEL は大文字右詰め5桁」。pytest 4本
- **F3 入力チェック**（holdoutペア）: 曖昧版「変な入力はじいて」/
  構造化版「email は RFC 風の正規表現、年齢は 0–150 の整数、名前は空文字禁止。
  違反は ValueError で理由を含める」。pytest 5本
- ペアは `f1_vague` / `f1_spec` のように別タスクIDにし、seed と tests/ を共有する

### G. Web画面（tasks_ui/ に隔離。ui-static 採点）

- **G1 カウンター**（低）: 「+・−・リセットの3ボタンと数値表示。+は青(#2563eb)、リセットは赤系」。
  機能: クリック3種の状態変化（4 assert）/ 構造: ボタン3つ+表示1つ・横並び（3チェック）/
  色: 2ボタンの背景色（2チェック）
- **G2 TODOリストUI**（中）: 「入力欄+追加ボタン、項目クリックで取り消し線、残件数表示」。
  機能6 assert / 構造4チェック / 色2チェック
- **G3 ダッシュボード再現**（高・holdout）: 正解サンプル HTML（ヘッダー+サイドバー+カード3枚）を
  リポジトリに同梱し、**2変種**で出題:
  - `g3_text`: 自然言語の説明のみで再現させる
  - `g3_image`: 正解サンプルのスクリーンショット PNG を渡して再現させる
  採点は同一（機能2 / 構造8 / 色4 チェック）。text と image のスコア差 =
  「画像モダリティの価値」の定量値。ローカルがテキスト専用モデルの場合、g3_image は
  クラウド専用タスクになる（ルーターの特徴量「画像の有無」の教師データ）

## 4. 実施方法

1. **arm**: mock（配管確認）→ local_only → cloud_only → router の順。同一タスク・同一指示
2. **反復**: 各3回（既定）。集計は中央値
3. **実行単位**: カテゴリごとに分割して回す（ログは追記式なので中断・再開自由）。
   ローカル系はいつでも・何度でも。クラウド系はレート制限に当たったら翌ウィンドウで続き
4. **UIタスク**: `tasks_ui/` に隔離し、専用ランナー（Playwright 起動）で回す。
   通常ベンチと混ぜない（実行時間の桁が違うため）
5. **課金モードの記録**: `CLOUD_BILLING` は env スナップショットに記録されるだけ。
   サブスクでもAPIでも同じスイートを同じ手順で回す
6. **システム改善の比較実験**: 比べたい構成（router_version・local_model 等）ごとに
   **同一 suite バージョンを丸ごと再実行**。env スナップショットがグループ化キーになる

## 5. 計測方法

### 共通4軸（全カテゴリ・runs.jsonl から）

精度 `tests_passed/tests_total` / API換算 `api_equiv_usd`（+実支払 `cost_usd`）/
時間 `wall_s` / クラウド消費 `cloud_calls`・`cloud_out_tok`

### カテゴリ固有の主指標

| カテゴリ | 主指標 | 副指標 |
|---|---|---|
| A 簡単編集 | 成功率 | — |
| B 調査 | チェック通過率 | 回答の文字数（冗長さ） |
| C バグ調査 | **recall**（発見/仕込み） | false_positive 数 |
| D 脆弱性調査 | **recall** + type 正解率 | false_positive 数 |
| E 機能追加 | テスト通過率 | turns（自立実行の迷走量） |
| F 曖昧ペア | **Δ = 構造化版 − 曖昧版**（成功率・turns・out_tok） | — |
| G Web画面 | 層別スコア（機能/構造/色） | text vs image のスコア差 |

### システム向上の時系列計測（このスイートの存在意義）

- 「suite v1 × arm × env（router_version, local_model）」でスコアを固定台帳化
- 例: ルーターを rule-v1 → 学習器 v2 に替えたら、**同じ suite v1 を再実行**して
  成功率・regret・クラウド消費の差分を見る。ものさしが固定だから差分がシステムの向上分
- holdout タスクのスコアだけを「公式スコア」とする（dev はチューニングで汚れるため）

## 6. task.yaml の拡張仕様（実装時に追加するフィールド）

```yaml
id: c1_cart_bugs
category: bughunt            # 既存: edit/lookup/... に bughunt/vulnhunt/ui/ambiguous を追加
tier: mid                    # low / mid / high（難易度。ルーター特徴量にもなる）
scoring: manifest-recall     # pytest / report-match / manifest-recall / ui-static
holdout: false               # true のタスクは学習データ生成（build_dataset）から除外
variant_of: null             # F/G の変種タスクが親IDを指す（f1_vague → f1）
modality: text               # text / image（imageは画像添付が必要 = ルーター特徴量）
answer_file: BUGS.md         # report系の解答ファイル名（pytest系は不要）
```

- `holdout: true` は `build_dataset.py` が自動で除外する（実装時に対応）
- suite の凍結は `tasks/SUITE-v1.yaml`（タスクIDの一覧+この設計書のバージョン）で行う

## 7. 実装の順序（別途着手）

1. A（簡単編集）3本 — 既存の pytest 方式のままで作れる。まずここで型を確立
2. B（調査）3本 — `answer_file` + report-match 採点器の実装が必要（小さい）
3. C/D（バグ・脆弱性）6本 — manifest-recall 採点器の実装が必要
4. E（機能追加）3本 — pytest 方式のまま。タスク作成の工数が本体
5. F（曖昧ペア）6本 — E2 等の seed を再利用。`variant_of` 対応
6. G（UI）3本+変種 — ui-static 採点器（Playwright）の実装が最大工数。最後
7. `tasks/SUITE-v1.yaml` を書いて凍結 → 全arm×3反復のベースライン計測 → 以後はこの台帳と比較
