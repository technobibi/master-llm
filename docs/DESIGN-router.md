# 設計書: ルーターのデータと判定（②ルーティングデータ + 天秤）

**状態**: 設計のみ。判定ロジック本体（kNN・スコア計算のコード）は所有者が書く領域。
ここは**確定した設計事項**（データ契約と枠組み）だけをまとめる。会話上の探索・雑談は含めない。
根拠となった議論: 2026-07。関連文書は §9。

## 0. 3種のデータの区別（確定）

- **① 計測**: タスク + 隠しテスト（答えを隠す）→ pass率・時間・トークン・コストを測る
- **② ルーティング**: ベクトル + 誰が成功したか + シグナル → 振り分けを学習。①から**導出・参照**
- **③ チューニング**: タスク + 答えを**見せる** → モデル訓練
- **①と③は必ず別データ**（テストの答えを訓練に使うとリーク=カンニング）
- **③はベクトル化して②に紐付けない**。②=ベクトル+成否（答え不要）、③=生テキストの答え（ベクトル不要）。
  ベクトルは③の**選別**（重複除去・弱点領域の優先）にのみ間接的に効く

## 1. ②の格納原則（確定）

- **保存は個別、判定は領域**: タスクを個別の点で貯め、判定時に kNN で近傍=領域として集約
- **事実は保存、判断は導出**: 「ローカルが最適か」を焼き込まない。両arm の生の数字を保存し、
  最適かどうかは**判定の瞬間に、その時の重みで計算**（重みは日々変わる＝枠の逼迫等）
- **②は①の派生＋参照**（`source_run_ids` で①を指す）。重複コピーしない
- **クラウド成否も記録**（判定の主軸はローカルだが、両方失敗の検出・regret・損得のため）。
  ただし**タダで手に入るとき（ベンチ）だけ**。本番でわざわざクラウドを走らせない

## 2. ②の1レコードが持つ項目（確定）

- タスクの**意味ベクトル**
- **規模特徴**: `n_seed_files`, `seed_bytes`（埋め込みで測れない大きさの軸）← 実装済み
- **ローカル成否・クラウド成否**（True/False でなく `tests_passed/total` の**割合**）
- **時間・トークン・枠消費**（天秤の生の数字）
- **反復間のバラつき**（曖昧さの推定）
- **雑版 vs 構造化版の成否差**（プロンプト品質の効き）
- **逆質問の有無 + ターン数**（指示コスト）
- **モデル版**（`local_model` / `agent_version`）← 実装済み。データ失効に使う

## 3. 天秤（判定スコア関数）の設計（確定）

線形和にしない。**段階（辞書順）＋ 仮想コスト**:

1. **成功を第1関門**（辞書順: 成功 > 時間 > 枠）
   - 両方成功 → 2へ ／ 片方だけ成功 → その側 ／ 両方失敗 → §5（入力を直す）
2. **両方成功なら仮想コストで比較**:
   - `cost(local) = w_time × 時間`
   - `cost(cloud) = w_time × 時間 + w_quota(残枠) × 枠消費`
   - 仮想コストが小さい方を選ぶ
3. **`w_quota` は残枠の関数**（枠が減るほど大きく＝quota-aware。枯渇時は遅くてもローカル）

- **割合の意味はタスク依存**: コード=白黒寄り（4/5でも壊れている）、調査/UI=連続量として意味あり
- **割合が開く第3の道**: カスケード（ローカルが安定して ~80% 以上 → 下書き、クラウドが仕上げ）

## 4. 失敗の原因帰属（確定）

単一の成否では原因を区別できない。**2×2 で帰属**（良いプロンプトで直る? × 強いモデルで直る?）:

| | 強いモデルで直る | 強いモデルでも直らない |
|---|---|---|
| **良いプロンプトで直る** | 境界的 | **プロンプト問題** → 構造化（核B） |
| **良いプロンプトでも直らない** | **能力問題** → 強いモデル/分解 | **困難/不可能** → 分解 or 正直に報告 |

- **シグナル（安い→高い）**: 反復のバラつき → 逆質問 → 両方失敗 → 構造化版が成功（決定的、Fカテゴリが測る）
- **プロンプト最適化スイッチは「プロンプト問題」象限だけで発火**（困難タスクに撃たない）

## 5. 指示コスト指標（確定）

二値（雑/構造化）より、**「合格までにどれだけ命令が要ったか」＝指示コスト（連続量）**を測る。

- 低い=曖昧を汲める（良い）／ 高い=手取り足取りが要る（構造化の価値が高い領域）。**結果から逆算＝客観**
- 測り方: **指示のはしご**（段階プロンプト、合格した最小レベル）or **対話の往復回数**
- **はしごの作り方**: 模範解は既知なので、**答えから逆算して段階的に指示を足す静的データ**を事前に用意する
- **構造化版は「要求は精密、実装は沈黙」**（答えを漏らすと「答えを渡せば解ける」を測ることになり実験が壊れる）
- **全部ペア化しない**。プロンプト感度は領域依存なので、領域ごとにサンプリングして補間

## 6. エスカレーションの階段（確定）

```
簡単     → ローカル（最安。裸で十分な超簡単は裸、それ以外はエージェント）
中       → ローカルエージェント ／ 無理ならクラウド
両方失敗 → 入力を直す:
           ├ バラつく   → 曖昧 → 「明確な指示を」と聞き返す（構造化=核B）
           └ 一貫して失敗 → 難しい/破綻 → 分解 or 正直に「無理」と報告
```

## 7. データ収集の方針（確定）

- **サブスク=金でなく枠**（`api_equiv_usd` は架空値、実支払は0）
- **ローカルは全数（無料）+ クラウドは意味のある一部だけ**（失敗分・境界付近・較正サンプル）＝能動学習
- **領域ごとに代表をサンプリングして補間**（全数をクラウドで走らせない）。領域の一貫性が低いところは多めに
- **①のタスクは公開SWEベンチを借りる**: HumanEval/MBPP（小・数を稼ぐ）→ BigCodeBench（中）→
  **SWE-bench Verified（大・複数ファイル＝規模の穴埋め）**。汚染対策に複数混ぜる

## 8. 確定した不足（今後の実装対象）

- **大規模・複数ファイルのタスクが無い**（現状 最大5ファイル・521バイト）← 最重要
  → **実行系は実装済み**（SWE-bench Lite、[DESIGN-swebench.md](DESIGN-swebench.md)）。残るは計測データの蓄積
- **タスク数が少ない**（現状25 → 数百へ、公開ベンチ取り込み）
- **反復が薄い**（現状3 → 5〜10。バラつき測定のため）
- **タスク埋め込みの記録が未実装**（`task_vectors`。②の意味ベクトル）
- **逆質問検出の強化 / 指示コストのはしご**（静的データ）
