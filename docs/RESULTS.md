## 2026-07-13 埋め込み領域のコンフィデンス（前回の悲観を更新）

**問い**: 「埋め込みは成否を分離しない（NN一致率50%, n=30）」は、単にデータ不足では？
**方法**: ローカル現行版256タスクを埋め込みで8領域にクラスタし、領域ごとのローカル成功率を測る
（個別のNN一致ではなく**領域平均**。所有者の提案＝過去n回の成功率をコンフィデンスとして貯める、に相当）。

**結果**: 領域ごとに成功率が **57%〜90%（33ポイント差）**。領域間SD=0.113 が
ノイズ由来の期待SD=0.087 を上回る（比 1.3倍）。n=30の頃と違い、**領域で成功率が
本当に違う兆候**が出た。全体成功率 71.1%。

**解釈（前回結論の訂正）**:
- 前回の「NN一致率50%」は **1回計測のコイン投げノイズに埋もれる厳しい指標**だった。
  **領域平均**は20〜60件でノイズを打ち消すため、下の真の率が浮かぶ＝所有者の方法が頑健。
- → 「埋め込み+領域コンフィデンスで振り分け」は成立の芽あり（DESIGN-router の天秤に合致）。
- **未確定**: 1反復のため領域内ノイズが差を水増しの恐れ。小領域(n≈20)はCI±10%。
  **3反復 + 全591問**で 33pt 差が生き残るか要確認。

## 2026-07-14 Qwen3.6-35B-A3B ベースライン完走（591問・1反復・モデル比較2本目）

計測途中に推論失速の汚染を検知し、37run を invalid_runs.jsonl で無効化→クリーン環境で再計測
（DESIGN-telemetry「無効runの注釈」の初適用）。

**成績**: 全体 **468/591 = 79.2%**（HumanEval **96.3%**、MBPP **72.6%**）、wall中央値 37s。

**vs 30B（75.5%）**: +3.7pt。両方成功416 / 3.6のみ52 / 30Bのみ30 / 両方失敗93。
HumanEvalで+10.9ptと圧倒する一方、**MBPPは+0.9ptでほぼ互角**。

**⚠️ 固有の欠陥 = 推論失速**: 特定タスクで生成が 0.2〜1.2 tok/s に落ち 600s キャップ到達
→ 失敗。**静かなマシンでも同一タスクで再現**（環境でなくタスク×ランタイム依存。
ツール呼び出し文法制約とサンプリングの衝突を疑う）。最終失速率 **4.9%（29問）**、
うち18問は30Bが平均20秒余で成功する問題。30Bの失速は602runで1件（0.2%）。
tok/s 自体は 3.6 の方が速い（中央値30.1 vs 25.6）— 遅いのではなく「時々止まる」。

**採用判断への含意**: 平均精度では 30B に勝つが、(a) 実務寄りMBPPでは互角、
(b) 失速でテール遅延が桁違いに悪い（20秒の問題に10分）、(c) 常駐+2GB 重い。
ローカル層に求める「速く安定した一次処理」としては 30B 優位の材料が多い。

## 2026-07-14 SWE-bench Lite 8問トリプル計測（1問 = 3.6→opus→Gemma4 直列・所有者設計）

**条件**: flask/requests/pytest/sphinx 各2問。max-turns 30 / max-wall 1800s / max-cost $2。
計測衛生: 全ローカルアーム前に LM Studio サーバ再起動（ゾンビ生成汚染を実測したため）。
タイムアウト系は「無効化→クリーン再計測1回まで、再現したら有効」ルール（計7run追い直し）。

**結果（テスト到達 / 公式解決◎ / キャップ超過解決○）**:

| instance | 30B | 3.6 | opus-4.8 | Gemma4 |
|---|---|---|---|---|
| flask-4045 | 41/52 | 50/52 | 51/52 | 50/52 |
| flask-4992 | - | 18/19 | ◎19/19 | 0/19 |
| requests-1963 | - | 0/119(空応答) | ◎119/119 | 0/119 |
| requests-2148 | - | 67/128 | ◎128/128 | 126/128 |
| pytest-11143 | - | 0/115(失速) | ◎115/115 | 0/115 |
| pytest-11148 | - | 0/131(破壊) | ○131/131($2.06) | 0/131 |
| sphinx-10325 | - | 0/6(欠陥×2回) | ○6/6(31turns) | 0/6 |
| sphinx-10451 | - | **80/82** | 79/82($2.72) | 0/82 |

**集計**: opus 公式4/8・技術6/8・到達率99%・$11.04（$1.38/問・実払$0・中央値95s）。
3.6 解決0/8・到達率43%（**健全動作時は3問で94%超**、sphinx-10451ではopus超え）。
Gemma4 解決0/8・到達率24%（7/8タイムアウト）。30Bはflask-4045のみ41/52。

**発見**:
1. **実リポで序列が確定**: 小タスク(94.2% vs 75.5%)の差が、実リポでは 6/8 vs 0/8 に開く。
   ただし敗因の大半は「最後の1-2テスト」= ゴールの完全性（R11テスト駆動分業の直接根拠）
2. **難問はクラウドも高くつく**: 2問でキャップ超過（$2.06/31turns/$2.72）。
   sphinx-10451 は3.6がopusを上回った初事例 — 難問の丸投げは枠効率が悪い（R3/R10材料）
3. **3.6の欠陥は実リポで増幅**: 失速・空応答・破壊編集で8問中4問が欠陥起因の失敗
   （小タスクでは4.9%だった）。長文脈が発火条件
4. **Gemma4はランタイム欠陥で計測不能**: mlx-lm の RotatingKVCache が prompt cache 不能
   → 毎ターン全文再処理（二乗税）→ 600s read-timeout。クリーンサーバでは 50/52・126/128 まで
   到達しており**認知能力は3.6同等**。GGUF+llama.cpp での再試験が公平な再挑戦
   （詳細: runs/investigations/20260714-swebench-triple/gemma4-timeout-analysis.md）
5. **計測方法論の副産物**: サーバ汚染（prefill 2700→16 tok/s の進行性劣化）を発見・定量化。
   全ローカルアーム前サーバ再起動をプロトコル化。調査一次資料はドシエに索引化

## 2026-07-14 重大訂正: 「失速欠陥」の正体はハーネスのバグ + Gemma4 GGUF 検証

**発見(所有者の「何かが間違ってるはず」が正解)**: agent.py のリクエストに max_tokens が
無く、モデルが EOS を出さない繰り返しループに入ると**無限生成** → クライアント600s
タイムアウト → サーバにゾンビ生成が残存し後続を汚染。`lms ps` で20分以上 GENERATING の
現行犯を確認。**「Qwen3.6/Gemma4 の失速・タイムアウトはモデル欠陥」という先の結論は
この計測器バグとの複合だった**。

**修正(b0d3f1c)**: AGENT_MAX_OUT_TOKENS 安全弁（既定0=従来挙動、設定時 env に記録）。
検証: 4回連続ハングしていた flask-4992 が 183s・30ターン・18/19 に。

**Gemma4 GGUF(llama.cpp)+安全弁4096 の8問再計測（タイムアウト0件）**:
flask-4045 0/52・flask-4992 **18/19**・requests-1963 0/119・requests-2148 0/128・
pytest-11143 **114/115**・pytest-11148 0/131・sphinx-10325 0/6・sphinx-10451 0/82

**Gemma4 の最終評価（正しく動く状態で）**:
- タイムアウト問題は完全解消（MLXキャッシュバグ回避 + 安全弁）
- 能力の閃きは本物（2/8で惜敗、うち pytest-11143 は 3.6 が2回全滅した問題）
- しかし**実行ごとの分散が極端**（同じ requests-2148 で 126/128 ⇔ 0/128）で、
  リポ破壊型の編集（全文上書きの失敗）が6/8を占める。敗因はもはや時間ではなく編集の信頼性
- 器v4候補（部分置換型編集ツール）で救済できる可能性はあるが、現行の器では3モデル中最弱

**次の課題**: 3.6 の「失速」失敗（SWE-bench 2問 + 公開ベンチ29問）を安全弁つきで再計測し、
真の実力値に補正すること。3モデル比較はその後に確定。
