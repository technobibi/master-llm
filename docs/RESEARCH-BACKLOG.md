# 研究バックログ（本線には載せない）

本線は「計測基盤」と「学習データ基盤」（→ [ARCHITECTURE.md](ARCHITECTURE.md)）。
ここにあるのは **データが貯まった後に試す研究テーマ** 。着手条件を満たすまで実装しない。

優先度: ★★★=本線の次にやる価値が高い / ★★=面白いが急がない / ★=いつか

---

## ★★★ R1. カスケード arm（タスクレベルの投機的実行）

**案**: ローカルが書き、失敗したらクラウドへ昇格する。さらに進めて
「ローカルが書き、クラウドは書かずにレビュー・修正だけする」構成にすると、
クラウドの出力トークン（遅くて高い方）を激減させられる。

**注意**: 昇格判定の信号に隠しテストを使うとリークになる。
使ってよいのは: 構文チェック / seed 同梱の公開スモークテスト / ローカル自身に書かせた self-test。

**着手条件**: local arm に実行→修正ループが入っていること（非対称性の解消が先）。

## ★★★ R2. 学習ルーター（ロジスティック回帰 → contextual bandit）

**案**: `router.jsonl` + ペアデータから「ローカルで成功するか」を予測する分類器を学習し、
`router.is_simple()` を差し替える。第1段はプロンプト特徴のロジスティック回帰で十分。
第2段は LinUCB / Thompson sampling にして、たまに探索的に逆側へ送り、
反実仮想データ（「ローカルに送ったらどうだったか」）を集め続ける。

**強み**: ベンチが全 arm を同一タスクで回すので、production ルーターが持てない
ペア（反実仮想）ラベルが最初から手に入る。

**着手条件**: routing データセット（DESIGN-dataset.md 参照）が 100 ペア以上。

## ★★★ R3. サブスク枠を意識したルーティング（quota-aware routing）

**案**: ルーティング研究はほぼ全部「クラウド=従量課金、$最小化」前提。
うちの構成では Claude はサブスクで限界費用ほぼゼロ、希少なのは **5時間ウィンドウのレート制限枠**。
目標関数を「$削減」から「1日の Claude 枠を限界価値が最も高いタスクに割り当てる」
ナップサック/スケジューリング問題に変える。ほぼ未開拓で、サブスクユーザー全員に刺さる。

**前提**: 計測基盤で枠消費の代理指標（cloud 呼び出し回数・out_tok）を記録していること。

## ★★ R4. 不確実性ベースの昇格（self-consistency 投票）

**案**: 生成前に分類するのではなく、ローカルで N 回サンプリングして答えが割れたらクラウドへ。
ローカルのトークンはタダなので投票コストはゼロ。

**研究角度**: 「小型モデルは自分ができないことを知っているか?」
ローカルの自己申告確信度 vs 実成功率で較正カーブを描く。個人データの較正はそれだけで記事になる。

## ★★ R5. 蒸留フライホイール（Claude が Qwen を教える）

**案**: クラウドが成功した (プロンプト, 最終diff) ペアを貯め、mlx-lm の LoRA で
ローカル Qwen を微調整。時間とともにローカル成功率が上がる=ルーターの閾値も動く生態系。
M5 Air 32GB で LoRA は現実的。nanoGPT に行く前の学習ステップとして最適。

**前提**: データ基盤が (prompt, diff) ペアを artifacts に保存していること（保存は本線）。

**確定済みの結論（2026-07-12 の議論）**:
- **公開SFTセットの「追い炊き」は期待値ほぼゼロ**: Qwen3-Coder は公開コードデータを
  学習し尽くした配布物。同じ分布を小規模LoRAで再投入しても汎用能力は上がらず、
  破滅的忘却で下がるリスクだけ残る。汎用強化の唯一の現実的経路は**新チェックポイントへの乗り換え**
  （report はモデル別集計に対応済み＝差し替え比較の基盤あり）
- 価値がある教材は「配布物に**入っていない**もの」だけ: 自分のタスク分布 + Opus の成功解
  （ミラー実行の副産物として蓄積中。共有規則上も「合成」に分類され安全）
- **着手条件の目安**: クラウド教師デモ 300件以上。最初の実験は Opus→7B 蒸留
  （32GB で 30B の LoRA は窮屈。7B 高速枠の底上げがルーター経済に効く）
- **配布の解答集（公開蒸留データ）の正しい使い道**: 能力の上積みではなく**壊れ止め**。
  少数の自前教材だけでLoRAすると破滅的忘却が起きるため、汎用分布の混合材として数千件混ぜる
  （主役=自前のOpus成功解、脇役=配布解答集。主客を逆にしない）。
  個人の範囲の規模限界の本当の答えは連合学習（DESIGN-learning-loop 種類2c・人が集まってから）
- 効果検証は凍結スイート + agent_version/モデル別集計で「上がったか壊れたか」を差分測定

## ★★ R6. Router regret（オラクル比較）を report に追加

**案**: ベンチは全 arm を回すのでタスクごとの最良 arm（オラクル）が計算できる。
「ルーターはオラクルに対して何%損しているか」はルーティング論文の標準指標。
`report.py` に数行足すだけ。実装が軽いので、計測基盤が固まり次第やってよい。

## ★ R7. エネルギー計測（第5の軸）

**案**: macOS の `powermetrics` でローカル推論のワット時を測る。
「ローカルは$0」という前提自体の検証。ルーティング評価で消費電力を報告する例はほぼない。
M5 実測値は希少データ。ついでに `tok/s` の熱ドリフト（連続実行での性能低下）も見える。

## ★ R8. プロンプト圧縮 / 文脈蒸留

**案**: クラウドへ送る前にローカルモデルがリポジトリ文脈を要約し、クラウド入力トークンを削減。
サブスク下では枠の節約になる。日本語圧縮の関心（言語による情報密度の差）ともつながる。

## ★ R9. 分解ルーティング（計画=クラウド、実行=ローカル）

**案**: タスク丸ごとではなくステップ単位でルーティング。クラウドが計画を立て、
ローカルがサブタスクを実行する plan-then-execute 構成。R1 カスケードの発展形。
