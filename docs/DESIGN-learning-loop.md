# 設計書: 学習ループとデータ共有（ベクトルルーター × モデル強化）

**目的**: 「振り分けを学習するルーター」と「実行能力を上げるモデル強化」を安全に噛み合わせる。
そのために必要な仕組み（データの賞味期限・探索・共有・実行隔離）を1か所にまとめる。
**状態**: 設計のみ。着手条件は各節に記す。中核ロジック（ルーター判定・訓練）は所有者が書く。

この文書は長い設計議論（2026-07）の結論を凍結したもの。実装前に必読。

## 0. 2つの学習は別物（混同しない）

| | システムの学習 | モデルの学習 |
|---|---|---|
| 何が賢くなる | オーケストレーター（振り分け） | モデル自身（実行能力） |
| 重み | 触らない | 更新する（LoRA/蒸留） |
| 手段 | ベクトル×規模で kNN／分類器 | 成功解でファインチューニング |

**振り分けは能力を上げない。振り分けを最適化するだけ。** 能力を上げるのはモデルの学習（蒸留）。
この2つは循環して回る（§1）。

## 1. フライホイール（振り分け → 蒸留 → 再計測）

```
ルーティング（今の得意領域で振り分け）
   ↓ 成功解を集める（build_dataset --kind sft）
LoRA蒸留（ローカルの実行能力を上げる。Claudeの成功解が教材）
   ↓ モデル版が上がる
再計測（凍結スイートを回し直し、新モデルの成功率を取り直す）
   ↓ 境界がローカル側へ動く
またルーティング（前より多くをローカルへ）
```

## 2. 自己成就予言の罠（最重要・素朴に繋ぐと壊れる）

**問題**: ルーターが「ローカルはXが苦手」と記録 → Xを送らない → 蒸留でXができるように
なっても、古い記録のせいで送られず、改善の証拠を永遠に得られない（router staleness）。

**原因**: 探索/活用の欠如 ＋ 概念ドリフト（モデルが変わると旧データが無効になる）。

**対策（4つ重ねる）**:
1. **モデル版でデータを紐付け**: 全ルーティングデータに `local_model` + `agent_version` を記録
   （env スナップショットに実装済み）。振り分けは**最新版のデータだけ**を見る。旧版は消さず無視。
2. **強化のたびに再計測**: 蒸留したら凍結スイートを回し直す。まず「前は失敗した領域」を優先再計測
   （フル再計測は高いので差分で。完全リセットの精神を軽く実現）。
3. **探索（バンディット）**: ルーターが「無理」と予測しても、たまにわざとローカルへ送り新証拠を取る。
   探索が無いとルーターは硬直する（→ R2）。
4. **カスケード保険**: 間違えても実行検証で拾える（→ R1）。

**着手条件**: 学習ルーター（R2）を実装するとき、この4点を最初から入れる。後付けは事故る。

## 3. 反実仮想の完全性（クラウド成功率も要る）

良い振り分けには「ローカルが成功するか」だけでなく「**クラウドが成功するか**」も要る。

- **ベンチモード（オフライン・今ここ）**: 全arm を同じタスクで回すのでペアで揃う（実装済み）。
  ベンチの価値は「本番では取れない完全な反実仮想が取れる」こと。
- **本番モード（オンライン）**: 選んだ側の結果しか観測できない（バンディットフィードバック）。
  → §2 の探索で埋める。
- 注意: **クラウドも常に成功ではない**。判定は「local: X%@$0 vs cloud: Y%@$コスト」の比較。
  クラウドすら苦戦するタスク＝「難しすぎ/曖昧すぎ」＝構造化（核B）か人間が要る、の別信号。

## 4. データ共有（生データなし・逆算不可・ウィルスなし）

「1人分ではデータが足りない」問題への答え。**共有したいデータの種類ごとに安全策が違う。**

### 種類1: ルーティングデータ（得意領域の地図）→ 逆算不可能な要約なら共有可

- 生ベクトルは vec2text で逆算されうる（機密コードが復元される）。だから**生ベクトルは送らない**。
- 送るのは: ベクトルを粗く量子化した**領域ID** + カテゴリ + 規模 + 成功/失敗 だけ。
  個々のタスクは復元不能、「どの領域か」だけ分かる（「暗号化版ベクトル」の現実的な形）。
- 集めると「この領域はローカル成功率が高い」という**地図の骨格**が、誰の生データも漏らさず共有できる。
- **着手条件**: オプトイン。送信前にサニタイズ。

### 種類2: チューニングデータ（モデルを賢くする教材）→ 最難関。急がない

チューニングは**生のコード（タスクと解）**を食う。要約では訓練できない＝種類1のトリックが使えない。
- (a) **公開・合成タスクのみ**: 実業務コードを集めない。安全だが実タスクの質は得られない。**当面これ**。
- (b) 明示合意 + 自動サニタイズ + **人間の最終レビュー**を通ったものだけプールへ。手間大。
- (c) **連合学習**: 生データを集めず、各端末で訓練した**重みの更新分だけ**集約（+差分プライバシー/
  セキュア集約）。生コードは1バイトも出ない。理想だが重い。**人が集まってから**。
- **正直な結論**: 実データのチューニング共有は事故時の被害（他人の機密コード漏洩）が甚大。
  (a) で始め、(c) に到達するまで実データ共有はしない。

### 種類3: 実行の安全性（ウィルス）→ 共有と独立に今すぐ必要

他人由来／自動生成のタスクを実行する以上、**サンドボックス化が絶対条件**。データ共有以前に、
今のハーネスも既にモデル生成コードを実行している点に注意。
- 隔離環境（コンテナ等）で実行 / ネットワーク遮断 / ファイルは作業ディレクトリ内に限定 /
  時間・リソース上限。
- 「データは決してコードとして実行しない。実行が必要なコードは隔離環境で完結させ母艦に触れさせない」。
- **着手条件**: 外部由来のタスクを1件でも受け入れる前に必須。

## 5. 実装順序（安全な階段）

1. 配布時に **公開・合成タスクの共通地図（骨格）** を同梱 → コールドスタート緩和
2. 各ユーザは自分の実タスクで地図を肉付け（ローカル・非共有）
3. 学習ルーター（R2）を §2 の4点込みで実装
4. sft データで LoRA 蒸留 → 凍結スイートで「実際に賢くなったか」を再計測（§1 のフライホイール）
5. サンドボックス化（§4-3）を入れてから、種類1の逆算不可データの任意共有
6. 人が集まったら連合学習（種類2c）
