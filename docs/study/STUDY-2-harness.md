# 勉強ノート2: この評価ハーネスは何をしていて、なぜその形なのか

master-llm の計測ルール1つ1つには理由がある。ルールを暗記するのではなく、
「外すと何が壊れるか」で理解する。これは実はML評価・実験計画の核心そのもので、
ここが分かればMLエンジニアの仕事の半分(モデルを作る前の評価設計)が分かる。

## 0. そもそも何をやっているのか(1段落で)

「ローカル+クラウドの組み合わせはClaude単独より得だ」という**主張**を、
感想ではなく**実験**で確かめようとしている。実験の形は医薬の治験と同じ:
同じ課題(タスク)を、条件(arm)だけ変えて複数回やらせ、結果(精度・$・時間)を比べる。
arm という名前は、治験の「投与群/対照群」を arm と呼ぶことから来ている。
`cloud_only` が対照群(現状のやり方)、`router` が新薬。

## 1. なぜテストを隠すのか — グッドハートの法則

「指標が目標になると、指標として機能しなくなる」(グッドハートの法則)。
テストをエージェントに見せると、問題を解く代わりに**テストだけ通る解**を書く
(ハードコードで期待値を返す等)。人間の受験勉強と同じで、過去問を丸暗記した子の
点数は実力を測っていない。

これはMLの最重要概念**「訓練とテストの分離」「リーク(leakage)」**の具体例。
モデルに見せた情報で評価してはいけない。この一点を体で理解していれば、
Kaggleでも論文でも実務でも、他人の「精度99%」を正しく疑えるようになる。

→ `runner.verify()` が実行後に初めてテストをコピーして回すのはこのため。
→ カスケード(R1)の昇格判定に隠しテストを使ってはいけない理由も同じ。
  昇格判定に使った瞬間、それは「モデルに見せた情報」になる。

## 2. なぜ予算キャップか — 外れ値と打ち切り

LLMは稀に暴走する(無限に修正し続ける等)。キャップなしだと、1回の暴走が
平均を破壊する。例: 9回が60秒、1回が3600秒 → 平均414秒。この数字は何も表さない。

対策は2つセット:
- **キャップ**(`task.yaml` の budget): 上限で打ち切り、失敗として数える
- **中央値**: 順番に並べた真ん中。外れ値に引きずられない

「平均は外れ値に弱い、中央値は頑健(robust)」は統計の基本装備。
「平均年収」がなぜ実感とズレるかと同じ話。

## 3. なぜ反復するのか — 確率的なものは1回では測れない

STUDY-1 §3 の通り、LLMの出力はくじ引き。1回の成功は実力かもしれないし運かもしれない。
コインを1回投げて表が出ても「このコインは必ず表が出る」とは言えないのと同じ。
反復して初めて「成功**率**」が言える。反復数が多いほど数字は信用できるが時間がかかる。
このトレードオフの調整が `--repeats`。

## 4. このハーネスの一番の武器 — 反実仮想(はんじつかそう)

「ローカルに送ったタスク、クラウドに送っていたらどうなっていたか?」
この「実際には起きなかった側の結果」を**反実仮想(counterfactual)**と呼ぶ。

本番運用のルーターは、選ばなかった側の結果を永遠に知れない。
だから「ルーターの判断は正しかったか」を後から検証できない。
このハーネスは**全armに同じタスクをやらせる**ので、両方の結果が揃う。
これがデータセット設計(DESIGN-dataset の routing ペア)の心臓部で、
production のログより質の高い教師データになる理由。

医療で言えば「同じ患者に薬ありと薬なしを両方試せる」という、
現実には不可能な理想実験を、タスクのコピーで実現している。

## 5. なぜログは追記のみ・生データ優先か — データは取り直せない

- **追記のみ(append-only)**: 書き換え可能なログは「後から都合よく直した」疑いを
  自分自身に対しても排除できない。会計帳簿と同じで、訂正も新しい行で行う。
- **生データ優先(raw > derived)**: 特徴量や集計は後で計算し直せるが、
  元のプロンプトや応答は保存していなければ永遠に消える。
  「後からこういう分析がしたくなった」は必ず起きる。派生値しか残していないと詰む。

→ 実装より先に artifacts 保存を優先せよ(DESIGN-dataset 実装ステップ1)の根拠。

## 6. なぜ cost_usd と api_equiv_usd を分けるのか — 測定単位の正直さ

サブスク下では `claude -p` が返す金額は**請求されない架空値**。
架空値と実払いを1つの列に混ぜると、「いくら節約した」という主張の根拠が崩れる。
将来API化したら api_equiv_usd 側がそのまま実請求額になる。
**単位や意味が違う数字は絶対に同じ列に入れない** — データ設計の鉄則。

## 7. 3軸+枠を「セットで」見る理由 — 単独指標は必ず嘘をつく

- 精度だけ見る → 全部クラウドに送れば勝ち(コスト無視)
- コストだけ見る → 全部ローカルに送れば勝ち(精度無視)
- 速度だけ見る → 何もしないのが最速

どれか1つを最適化すると他が犠牲になる(トレードオフ)。だから主張は常に
「**精度を保ったまま**コストを何%減らせたか」の形にする。
ML論文が精度と計算量を並記するのも、これと同じ規律。
