# 勉強ノート1: LLMの仕組み — このプロジェクトの理解に必要な分だけ

一般論の教科書ではなく、master-llm のコードと設計判断に全部つなげて説明する。
数式は使わない。最後の「自分の言葉で答える問い」に答えられたら合格。

## 1. LLMは「次のトークン予測機」でしかない

LLMがやっていることは1つだけ: **文章の続きとして一番ありそうな次の1トークンを予測する**。
それを1トークンずつ何百回も繰り返して文章が出てくる。計画も理解も「ない」が、
膨大なテキストの統計を圧縮した結果、あるかのように振る舞う。

**トークン**は「文字」でも「単語」でもない中間の単位。英語だと1トークン≒0.75単語、
日本語だと1文字が1〜3トークンに割れることが多い。

→ **自分のプロジェクトとの接続**: `runs.jsonl` の `in_tok` / `out_tok` はこの単位を
数えている。課金も速度もすべてトークン単位。あなたが感じていた「日本語は圧縮率が悪い」
問題の正体の半分はトークナイザが英語中心に作られていること。

## 2. コンテキストウィンドウ: LLMに「記憶」はない

LLMは毎回、**渡されたテキスト全部**(=コンテキスト)を読み直して次のトークンを出す。
会話しているように見えるのは、ツール側が過去のやり取りを全部つなげて毎回送り直しているから。

だからエージェント(Claude Code)がターンを重ねると:
- ターン1: 指示(1,000トークン)を読む
- ターン2: 指示+ファイル内容+自分の応答(5,000トークン)を**全部読み直す**
- ターン3: さらに増えた全部を読み直す…

→ **接続**: 入力トークンがターンごとに雪だるま式に増える。これが
DESIGN-telemetry の「ターン単位のusageを取りたい」理由。「曖昧な指示は迷走して
コンテキスト成長が速い」という仮説は、この曲線を見ないと検証できない。

**プロンプトキャッシュ**: 毎回同じ前半部分を読み直すのは無駄なので、APIは
「前回と同じ部分」の計算結果を保存して再利用する。キャッシュから読んだ入力は
約1/10の価格。エージェント動作では入力の大半がキャッシュ読みになる。
→ だから calls.jsonl に `cache_read_tok` を足さないとAPIコスト予測が数倍ズレる。

## 3. 出力は確率的: 同じ質問でも毎回違う

次のトークンは「確率が一番高いもの」を必ず選ぶのではなく、確率に応じて
**くじ引き**する(サンプリング)。くじの偏り具合を調整するのが **temperature**。
0に近いほど毎回同じ答え、高いほどばらける。

→ **接続**: `clients.py` でローカルに `temperature: 0.2` を渡しているのはこれ。
そして「LLMは確率的だから `--repeats 3` 以上・中央値で見る」という計測ルールの根拠。
1回の成功/失敗は運かもしれない。

## 4. なぜ大きいモデルは賢いか、なぜ30BがM5 Airで動くか

パラメータ(モデル内部の数値)が多いほど、学習データの統計を細かく覚えられる。
Claude級は推定で数千億〜、あなたのローカルは30B(300億)。この差が
「エージェント的な複雑タスクの成功率の差」になる。

ただし Qwen3-Coder-30B-**A3B** の A3B は "Active 3B" の意味。**MoE(Mixture of Experts)**
という構造で、300億パラメータのうち毎トークン30億分しか計算しない。
賢さは30B寄り、速度とメモリ帯域の消費は3B寄り。
→ M5 Air 32GB で実用速度が出るのはこの構造のおかげ。

## 5. コストと速度の構造: 入力と出力は別物

- **入力トークン**: 一括で並列処理できる → 速くて安い
- **出力トークン**: 1個ずつ順番にしか作れない → 遅くて高い(APIだと入力の5倍等)

ローカルの生成速度(`tok_per_s`)を決めるのは主に**メモリ帯域**(GPUがパラメータを
読み出す速さ)。Macが連続で推論すると発熱で帯域が絞られ、tok/s が落ちる。
→ calls.jsonl に `tok_per_s` を入れたのはこの熱ドリフトを検出するため(R7)。

→ **ルーターへの接続**: 「クラウドは書かずにレビューだけさせる」(R1)が効くのは、
レビューは出力トークンが少ないから。高い方のトークンを節約する発想。

## 6. 「エージェント」とは: LLMをループの中に入れたもの

素のLLMは1回テキストを返して終わり。エージェントは
**LLM → ツール実行(ファイル読む・コード実行) → 結果をコンテキストに追加 → またLLM**
というループ。`claude -p` の `num_turns` はこのループ回数。

→ **接続**: 現状の非対称性問題の正体。cloud arm はこのループを持ち、
local arm は1回出力して終わり(`applier.py` で貼るだけ)。モデルの賢さの差ではなく
**ループの有無の差**を測ってしまっている。だから local にも最低限のループを足す必要がある。
